### 题目1：String 内存指向
> String s1 = "java"; 和 String s2 = new String("java");，== 和 equals 的结果？

**我的初答**：==的结果是ture,equals不懂
<details>
<summary><strong>点击展开标准解析</strong></summary>

- **`s1 == s2` 为 `false`**：`s1` 指向**字符串常量池**（String Pool）中的地址；`s2` 指向**堆内存**（Heap）中新创建的对象地址。两者内存地址不同。
- **`s1.equals(s2)` 为 `true`**：`String` 类重写了 `equals()`，逐个比较底层 `char[]` 数组的字符序列是否相同。
- **进阶加分项**：如果想让 `s2 == s1` 返回 `true`，可以调用 `s2.intern()`，该方法会强制将堆中的字符串引用指向常量池中已有的对象。
</details>
**错漏点**：并不了解==与equals的区别点

### 题目2：ArrayList vs LinkedList
> 分别适合什么场景？

**我的初答**：不知道
<details>
<summary><strong>点击展开标准解析</strong></summary>

- **`ArrayList`（底层数组）**：**查询修改快**（O(1) 随机访问），**中间插入/删除慢**（需 `System.arraycopy` 移动元素）。适合**读多写少**的场景（实际项目占比超 95%）。
- **`LinkedList`（底层双向链表）**：**头部/尾部插入删除快**（O(1) 改指针），**查询慢**（O(n) 需从头遍历）。适合**头部频繁插入**的场景（如队列）。
- **致命误区**：很多人误以为 `LinkedList` 所有增删都快，但在**中间位置增删**时，两者都需先遍历到该位置，`ArrayList` 遍历更快，所以反而胜出。
- **加分项**：提及 `ArrayList` 扩容机制为 **1.5 倍**（JDK 8），建议开发中预估容量，使用 `new ArrayList<>(initialCapacity)` 避免频繁扩容。
</details>
**错漏点**：需要学习ArrayList和LinkedList的相关知识

### 题目3：HashMap 哈希寻址
> 如何确定下标？哈希冲突怎么办？

**我的初答**：不知道
<details>
<summary><strong>点击展开标准解析</strong></summary>

**执行流程分三步**：

1.  **算哈希（扰动）**：调用 `key.hashCode()` 得到原始哈希，再执行 `h ^ (h >>> 16)`（高16位与低16位异或）。目的是让高位特征参与低位运算，**减少哈希碰撞**。
2.  **算下标（寻址）**：使用 `(n - 1) & hash` 计算数组下标（`n` 为数组长度，必须是 2 的幂）。此处用位运算代替取模 `%`，**效率更高**。
3.  **冲突解决（拉链法）**：若下标相同，先比较 `==` 地址，再比较 `equals()` 内容。若 key 完全相等则覆盖旧值；否则以**链表**形式挂载（JDK 8 尾部插入）。

- **进阶升级（树化）**：当链表长度 **> 8** 且数组长度 **≥ 64** 时，链表转为**红黑树**（O(log n)）。若数组长度 < 64，优先扩容而非树化，防止因数组过小导致的哈希碰撞。
</details>
**错漏点**：需学习哈希寻址的相关知识

### 题目1：重载（Overload）与重写（Override）的区别
> 请简述两者的区别，并说明它们在编译期和运行期的表现差异？

<details>
<summary><strong>点击展开标准解析</strong></summary>

- **重载（Overload）**：**同一类中**，方法名相同，**参数列表不同**（类型、个数、顺序）。与返回值无关。属于**编译期多态**（静态绑定），编译时根据传入参数决定调哪个。
- **重写（Override）**：**子类继承父类**，方法名、参数、返回值完全相同，访问修饰符不能更严格。属于**运行期多态**（动态绑定），运行时根据对象实际类型决定调哪个。
- **高频追问**：构造器能重写吗？**不能**（构造器不能被继承）。
</details>

**我的初答**：重载Overload是同一类下的相同方法,但参数列表不同 重写Override不了解
**错漏点**：Overload与返回值无关

### 题目2：equals() 与 hashCode() 的契约
> 为什么重写 `equals()` 时必须重写 `hashCode()`？只重写 equals 在 HashMap 中会出什么 Bug？

<details>
<summary><strong>点击展开标准解析</strong></summary>

- **核心契约**：两个对象如果 `equals()` 返回 `true`，则 `hashCode()` **必须相等**。如果 `equals` 返回 `false`，`hashCode` 可以相等（允许冲突）也可以不等（提高性能）。
- **只重写 equals 的 Bug**：假设自定义 `Student` 类，`id` 相同视为同一人。只重写 `equals` 不重写 `hashCode`，放入 `HashMap` 作为 key 时，`put` 会先算 `hashCode` 定位桶，再 `equals` 找值。两个相同 `id` 的对象 `hashCode` 不同，会被**放到不同桶里**，导致 `map.get(student1)` 拿不到 `student2` 存进去的值，**逻辑上相等的对象在集合中变成了两个键**，造成内存泄漏和业务错误。
- **加分项**：`HashSet` 底层就是 `HashMap`，同样会中招。
</details>

### 题目3：sleep() 与 wait() 的区别
> `Thread.sleep()` 和 `Object.wait()` 有什么区别？在锁释放和 CPU 占用上有何不同？

<details>
<summary><strong>点击展开标准解析</strong></summary>

| 对比维度 | Thread.sleep() | Object.wait() |
| :--- | :--- | :--- |
| **所属** | Thread 类的静态方法 | Object 类的实例方法 |
| **锁释放** | **不释放** 持有的对象锁 | **释放** 持有的对象锁 |
| **唤醒方式** | 时间到自动醒，或 `interrupt()` | 必须被 `notify()` / `notifyAll()` 唤醒 |
| **使用前提** | 任何地方都能调用 | **必须在 synchronized 块内**调用 |
| **CPU 占用** | 不释放锁，其他线程进不来；但线程本身不占用 CPU | 释放锁，让其他线程竞争，自身进入等待队列 |

- **核心一句话**：`sleep` 抱着锁睡，到点自己醒；`wait` 松开锁睡，等人叫醒。
</details>

### 题目1：final、finally、finalize 的区别
> 这三个关键字/机制分别是什么？各自的使用场景？

<details>
<summary><strong>点击展开标准解析</strong></summary>

- **final（关键字）**：
    - 修饰**类**：不可被继承（如 `String`）
    - 修饰**方法**：不可被子类重写
    - 修饰**变量**：基本类型值不可变，引用类型不可指向新对象（但对象内容可变）
- **finally（代码块）**：
    - 与 `try-catch` 配合，**无论是否发生异常，都会执行**（除非 JVM 提前退出）
    - 通常用于释放资源（关闭文件流、数据库连接等）
- **finalize（方法）**：
    - `Object` 类的方法，GC 回收对象前会调用（已废弃，JDK 9 标记为弃用）
    - **不推荐使用**：调用时机不确定，执行不可靠，容易引起性能问题
- **一句话总结**：final 是“不变”，finally 是“必须执行”，finalize 是“临终遗言（已过时）”。
</details>

### 题目2：String、StringBuilder、StringBuffer 的区别
> 分别适用于什么场景？哪个线程安全？

<details>
<summary><strong>点击展开标准解析</strong></summary>

| 对比维度 | String | StringBuilder | StringBuffer |
| :--- | :--- | :--- | :--- |
| **可变性** | **不可变**（final char[]） | **可变** | **可变** |
| **线程安全** | 安全（不可变天然安全） | **不安全** | **安全**（方法加 `synchronized`） |
| **性能** | 拼接时创建新对象，**最慢** | **最快** | 比 StringBuilder 稍慢（有锁开销） |
| **使用场景** | 少量拼接或字符串常量 | **单线程**下大量拼接 | **多线程**下大量拼接（极少用） |

- **高频追问**：`String s = "a" + "b" + "c";` 编译后会被优化为 `"abc"`，不创建中间对象。但循环内拼接，编译后会变成 `StringBuilder.append()`，**建议循环内手动用 StringBuilder 避免每次 new**。
- **一句话总结**：开发中 **90% 的场景用 StringBuilder**（单线程），StringBuffer 基本被淘汰，String 只用于常量。

### 题目3：检查型异常（Checked）与非检查型异常（Unchecked）的区别
> Java 中异常分为哪两大类？`RuntimeException` 属于哪一类？分别举例说明。

<details>
<summary><strong>点击展开标准解析</strong></summary>

- **检查型异常（Checked Exception）**：
    - 继承自 `Exception` 但**不继承** `RuntimeException`
    - 编译器**强制要求**处理（`try-catch` 或 `throws`）
    - **典型例子**：`IOException`、`SQLException`、`ClassNotFoundException`
- **非检查型异常（Unchecked Exception）**：
    - 继承自 `RuntimeException`
    - 编译器**不强制**处理，运行时才会抛出
    - **典型例子**：`NullPointerException`、`ArrayIndexOutOfBoundsException`、`ArithmeticException`、`ClassCastException`
- **设计理念**：Checked 代表可恢复的异常（如文件不存在），调用者应妥善处理；Unchecked 代表编程错误（如空指针），应在代码层面避免。
- **高频追问**：Spring 事务管理中，默认 **只对 Unchecked 异常回滚**，Checked 异常不会触发回滚。
</details>
